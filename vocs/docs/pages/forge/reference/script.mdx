## forge script

Run a smart contract as a script, building transactions that can be sent onchain

```bash
$ forge script --help
```

```txt
Usage: forge script [OPTIONS] &lt;PATH&gt; [ARGS]...

Arguments:
  &lt;PATH&gt;
          The contract you want to run. Either the file path or contract name.
          
          If multiple contracts exist in the same file you must specify the
          target contract with --target-contract.

  [ARGS]...
          Arguments to pass to the script function

Options:
  -j, --threads &lt;THREADS&gt;
          Number of threads to use. Specifying 0 defaults to the number of
          logical cores
          
          [aliases: --jobs]

      --target-contract &lt;CONTRACT_NAME&gt;
          The name of the contract you want to run
          
          [aliases: --tc]

  -s, --sig &lt;SIG&gt;
          The signature of the function you want to call in the contract, or raw
          calldata
          
          [default: run()]

      --priority-gas-price &lt;PRICE&gt;
          Max priority fee per gas for EIP1559 transactions
          
          [env: ETH_PRIORITY_GAS_PRICE=]

      --legacy
          Use legacy transactions instead of EIP1559 ones.
          
          This is auto-enabled for common networks without EIP1559.

      --broadcast
          Broadcasts the transactions

      --batch-size &lt;BATCH_SIZE&gt;
          Batch size of transactions.
          
          This is ignored and set to 1 if batching is not available or `--slow`
          is enabled.
          
          [default: 100]

      --skip-simulation
          Skips on-chain simulation

  -g, --gas-estimate-multiplier &lt;GAS_ESTIMATE_MULTIPLIER&gt;
          Relative percentage to multiply gas estimates by
          
          [default: 130]

      --unlocked
          Send via `eth_sendTransaction` using the `--sender` argument or
          `$ETH_FROM` as sender

      --resume
          Resumes submitting transactions that failed or timed-out previously.
          
          It DOES NOT simulate the script again and it expects nonces to have
          remained the same.
          
          Example: If transaction N has a nonce of 22, then the account should
          have a nonce of 22, otherwise it fails.

      --multi
          If present, --resume or --verify will be assumed to be a multi chain
          deployment

      --debug
          Open the script in the debugger.
          
          Takes precedence over broadcast.

      --dump &lt;PATH&gt;
          Dumps all debugger steps to file

      --slow
          Makes sure a transaction is sent, only after its previous one has been
          confirmed and succeeded

      --non-interactive
          Disables interactive prompts that might appear when deploying big
          contracts.
          
          For more info on the contract size limit, see EIP-170:
          &lt;https://eips.ethereum.org/EIPS/eip-170&gt;

      --disable-code-size-limit
          Disables the contract size limit during script execution

      --etherscan-api-key &lt;KEY&gt;
          The Etherscan (or equivalent) API key
          
          [env: ETHERSCAN_API_KEY=]

      --etherscan-api-version &lt;VERSION&gt;
          The Etherscan API version
          
          [env: ETHERSCAN_API_VERSION=]

      --verify
          Verifies all the contracts found in the receipts of a script, if any

      --with-gas-price &lt;PRICE&gt;
          Gas price for legacy transactions, or max fee per gas for EIP1559
          transactions, either specified in wei, or as a string with a unit
          type.
          
          Examples: 1ether, 10gwei, 0.01ether
          
          [env: ETH_GAS_PRICE=]

      --timeout &lt;TIMEOUT&gt;
          Timeout to use for broadcasting transactions
          
          [env: ETH_TIMEOUT=]

  -h, --help
          Print help (see a summary with '-h')

Display options:
  -v, --verbosity...
          Verbosity level of the log messages.
          
          Pass multiple times to increase the verbosity (e.g. -v, -vv, -vvv).
          
          Depending on the context the verbosity levels have different meanings.
          
          For example, the verbosity levels of the EVM are:
          - 2 (-vv): Print logs for all tests.
          - 3 (-vvv): Print execution traces for failing tests.
          - 4 (-vvvv): Print execution traces for all tests, and setup traces
          for failing tests.
          - 5 (-vvvvv): Print execution and setup traces for all tests,
          including storage changes.

  -q, --quiet
          Do not print log messages

      --json
          Format log messages as JSON

      --color &lt;COLOR&gt;
          The color of the log messages

          Possible values:
          - auto:   Intelligently guess whether to use color output (default)
          - always: Force color output
          - never:  Force disable color output

Cache options:
      --force
          Clear the cache and artifacts folder and recompile

Build options:
      --no-cache
          Disable the cache

      --dynamic-test-linking
          Enable dynamic test linking

      --skip &lt;SKIP&gt;...
          Skip building files whose names contain the given filter.
          
          `test` and `script` are aliases for `.t.sol` and `.s.sol`.

Linker options:
      --libraries &lt;LIBRARIES&gt;
          Set pre-linked libraries
          
          [env: DAPP_LIBRARIES=]

Compiler options:
      --ignored-error-codes &lt;ERROR_CODES&gt;
          Ignore solc warnings by error code

      --deny-warnings
          Warnings will trigger a compiler error

      --no-auto-detect
          Do not auto-detect the `solc` version

      --use &lt;SOLC_VERSION&gt;
          Specify the solc version, or a path to a local solc, to build with.
          
          Valid values are in the format `x.y.z`, `solc:x.y.z` or
          `path/to/solc`.

      --offline
          Do not access the network.
          
          Missing solc versions will not be installed.

      --via-ir
          Use the Yul intermediate representation compilation pipeline

      --use-literal-content
          Changes compilation to only use literal content and not URLs

      --no-metadata
          Do not append any metadata to the bytecode.
          
          This is equivalent to setting `bytecode_hash` to `none` and
          `cbor_metadata` to `false`.

      --ast
          Includes the AST as JSON in the compiler output

      --evm-version &lt;VERSION&gt;
          The target EVM version

      --optimize [&lt;OPTIMIZE&gt;]
          Activate the Solidity optimizer
          
          [possible values: true, false]

      --optimizer-runs &lt;RUNS&gt;
          The number of runs specifies roughly how often each opcode of the
          deployed code will be executed across the life-time of the contract.
          This means it is a trade-off parameter between code size (deploy cost)
          and code execution cost (cost after deployment). An `optimizer_runs`
          parameter of `1` will produce short but expensive code. In contrast, a
          larger `optimizer_runs` parameter will produce longer but more gas
          efficient code

      --extra-output &lt;SELECTOR&gt;...
          Extra output to include in the contract's artifact.
          
          Example keys: evm.assembly, ewasm, ir, irOptimized, metadata
          
          For a full description, see
          &lt;https://docs.soliditylang.org/en/v0.8.13/using-the-compiler.html#input-description&gt;

      --extra-output-files &lt;SELECTOR&gt;...
          Extra output to write to separate files.
          
          Valid values: metadata, ir, irOptimized, ewasm, evm.assembly

Project options:
  -o, --out &lt;PATH&gt;
          The path to the contract artifacts folder

      --revert-strings &lt;REVERT&gt;
          Revert string configuration.
          
          Possible values are "default", "strip" (remove), "debug"
          (Solidity-generated revert strings) and "verboseDebug"

      --build-info
          Generate build info files

      --build-info-path &lt;PATH&gt;
          Output path to directory that build info files will be written to

      --root &lt;PATH&gt;
          The project's root path.
          
          By default root of the Git repository, if in one, or the current
          working directory.

  -C, --contracts &lt;PATH&gt;
          The contracts source directory

  -R, --remappings &lt;REMAPPINGS&gt;
          The project's remappings

      --remappings-env &lt;ENV&gt;
          The project's remappings from the environment

      --cache-path &lt;PATH&gt;
          The path to the compiler cache

      --lib-paths &lt;PATH&gt;
          The path to the library folder

      --hardhat
          Use the Hardhat-style project layout.
          
          This is the same as using: `--contracts contracts --lib-paths
          node_modules`.
          
          [aliases: --hh]

      --config-path &lt;FILE&gt;
          Path to the config file

Wallet options - raw:
  -a, --froms [&lt;ADDRESSES&gt;...]
          The sender accounts for transactions when using local signers (private
          keys, keystores, hardware wallets, etc.).
          
          These addresses are derived from the provided private keys/signers and
          specify which accounts to use for signing transactions locally. This
          is different from --unlocked which uses eth_sendTransaction with
          pre-unlocked accounts on the RPC.
          
          [env: ETH_FROM=]

  -i, --interactives &lt;NUM&gt;
          Open an interactive prompt to enter your private key.
          
          Takes a value for the number of keys to enter.
          
          [default: 0]

      --private-keys &lt;RAW_PRIVATE_KEYS&gt;
          Use the provided private keys

      --private-key &lt;RAW_PRIVATE_KEY&gt;
          Use the provided private key

      --mnemonics &lt;MNEMONICS&gt;
          Use the mnemonic phrases of mnemonic files at the specified paths

      --mnemonic-passphrases &lt;PASSPHRASE&gt;
          Use a BIP39 passphrases for the mnemonic

      --mnemonic-derivation-paths &lt;PATH&gt;
          The wallet derivation path.
          
          Works with both --mnemonic-path and hardware wallets.

      --mnemonic-indexes &lt;INDEXES&gt;
          Use the private key from the given mnemonic index.
          
          Can be used with --mnemonics, --ledger, --aws and --trezor.
          
          [default: 0]

Wallet options - keystore:
      --keystore &lt;PATHS&gt;
          Use the keystore by its filename in the given folder
          
          [env: ETH_KEYSTORE=]
          [aliases: --keystores]

      --account &lt;ACCOUNT_NAMES&gt;
          Use a keystore from the default keystores folder
          (~/.foundry/keystores) by its filename
          
          [env: ETH_KEYSTORE_ACCOUNT=]
          [aliases: --accounts]

      --password &lt;PASSWORDS&gt;
          The keystore password.
          
          Used with --keystore.

      --password-file &lt;PATHS&gt;
          The keystore password file path.
          
          Used with --keystore.
          
          [env: ETH_PASSWORD=]

Wallet options - hardware wallet:
  -l, --ledger
          Use a Ledger hardware wallet

  -t, --trezor
          Use a Trezor hardware wallet

Wallet options - remote:
      --aws
          Use AWS Key Management Service

      --gcp
          Use Google Cloud Key Management Service

EVM options:
  -f, --fork-url &lt;URL&gt;
          Fetch state over a remote endpoint instead of starting from an empty
          state.
          
          If you want to fetch state from a specific block number, see
          --fork-block-number.
          
          [aliases: --rpc-url]

      --fork-block-number &lt;BLOCK&gt;
          Fetch state from a specific block number over a remote endpoint.
          
          See --fork-url.

      --fork-retries &lt;RETRIES&gt;
          Number of retries.
          
          See --fork-url.

      --fork-retry-backoff &lt;BACKOFF&gt;
          Initial retry backoff on encountering errors.
          
          See --fork-url.

      --no-storage-caching
          Explicitly disables the use of RPC caching.
          
          All storage slots are read entirely from the endpoint.
          
          This flag overrides the project's configuration file.
          
          See --fork-url.

      --initial-balance &lt;BALANCE&gt;
          The initial balance of deployed test contracts

      --sender &lt;ADDRESS&gt;
          The address which will be executing tests/scripts

      --ffi
          Enable the FFI cheatcode

      --always-use-create-2-factory
          Use the create 2 factory in all cases including tests and
          non-broadcasting scripts

      --create2-deployer &lt;ADDRESS&gt;
          The CREATE2 deployer address to use, this will override the one in the
          config

Fork config:
      --compute-units-per-second &lt;CUPS&gt;
          Sets the number of assumed available compute units per second for this
          provider
          
          default value: 330
          
          See also --fork-url and
          &lt;https://docs.alchemy.com/reference/compute-units#what-are-cups-compute-units-per-second&gt;

      --no-rpc-rate-limit
          Disables rate limiting for this node's provider.
          
          See also --fork-url and
          &lt;https://docs.alchemy.com/reference/compute-units#what-are-cups-compute-units-per-second&gt;
          
          [aliases: --no-rate-limit]

Executor environment config:
      --code-size-limit &lt;CODE_SIZE&gt;
          EIP-170: Contract code size limit in bytes. Useful to increase this
          because of tests. By default, it is 0x6000 (~25kb)

      --chain &lt;CHAIN&gt;
          The chain name or EIP-155 chain ID
          
          [aliases: --chain-id]

      --gas-price &lt;GAS_PRICE&gt;
          The gas price

      --block-base-fee-per-gas &lt;FEE&gt;
          The base fee in a block
          
          [aliases: --base-fee]

      --tx-origin &lt;ADDRESS&gt;
          The transaction origin

      --block-coinbase &lt;ADDRESS&gt;
          The coinbase of the block

      --block-timestamp &lt;TIMESTAMP&gt;
          The timestamp of the block

      --block-number &lt;BLOCK&gt;
          The block number

      --block-difficulty &lt;DIFFICULTY&gt;
          The block difficulty

      --block-prevrandao &lt;PREVRANDAO&gt;
          The block prevrandao value. NOTE: Before merge this field was mix_hash

      --block-gas-limit &lt;GAS_LIMIT&gt;
          The block gas limit
          
          [aliases: --gas-limit]

      --memory-limit &lt;MEMORY_LIMIT&gt;
          The memory limit per EVM execution in bytes. If this limit is
          exceeded, a `MemoryLimitOOG` result is thrown.
          
          The default is 128MiB.

      --disable-block-gas-limit
          Whether to disable the block gas limit checks
          
          [aliases: --no-gas-limit]

      --isolate
          Whether to enable isolation of calls. In isolation mode all top-level
          calls are executed as a separate transaction in a separate EVM
          context, enabling more precise gas accounting and transaction state
          changes

      --odyssey
          Whether to enable Odyssey features

      --retries &lt;RETRIES&gt;
          Number of attempts for retrying verification
          
          [default: 5]

      --delay &lt;DELAY&gt;
          Optional delay to apply in between verification attempts, in seconds
          
          [default: 5]

Verifier options:
      --verifier &lt;VERIFIER&gt;
          The contract verification provider to use
          
          [default: sourcify]

          Possible values:
          - etherscan
          - sourcify
          - blockscout
          - oklink
          - custom:     Custom verification provider, requires compatibility
            with the Etherscan API

      --verifier-api-key &lt;VERIFIER_API_KEY&gt;
          The verifier API KEY, if using a custom provider
          
          [env: VERIFIER_API_KEY=]

      --verifier-url &lt;VERIFIER_URL&gt;
          The verifier URL, if using a custom provider
          
          [env: VERIFIER_URL=]

      --verifier-api-version &lt;VERIFIER_API_VERSION&gt;
          The verifier API version, if using a custom provider
          
          [env: VERIFIER_API_VERSION=]
```
